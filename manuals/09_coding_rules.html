Antigravity直伝・コーディングの鉄則

「修正の無限ループ」から抜け出すための15の鉄則

← マニュアル一覧に戻る

🛡️ これはなに？

複雑なデザイン（C-Limber等）に挑戦した際に直面した「コードが複雑化して修正できなくなる」問題を解決するための、Antigravityとの約束事（鉄の掟）です。
模写だけでなく、オリジナル開発でもこのルールを守ることで、プロ品質の保守性を担保します。

1. モバイルファースト（小は大を兼ねない！）

「まずスマホ版のデザインを先に書く」。これだけでコード量は半分になります。

NG: Webファースト

デスクトップ版を先に作ると、スマホ版で「打ち消し」のコードが大量に必要になります。

padding: 0; border: none; flex-direction: column;

↑こんなコードが増えたら要注意です。

OK: モバイルファースト

スマホ版をベースにすれば、デスクトップ版は「追加」だけで済みます。

「スマホでシンプルな形を作る」→「画面が広かったら横並びにする（@media）」

2. !important は「敗北宣言」

!important を使いたくなったら、それはCSSの設計が破綻しかけている合図です。

NG

width: 100% !important;

これを使うと、後で「やっぱり変更したい」と思った時に、誰も修正できなくなります。

OK

使う前に自問自答する：
「HTMLのクラス名を見直せないか？」
「詳細度（セレクタの強さ）を整理できないか？」

3. 「絶対配置（absolute）」は飾りだけ

コンテンツ（文字や画像）を position: absolute で無理やり配置するのはやめましょう。

NG

本文や見出しなどの「文章」をabsoluteで固定する。
→ 文字数が増えたり、スマホで見たりした瞬間に崩壊します。

OK

中身の配置は必ず Flexbox か Grid を使います。
absolute は「背景の赤い丸」や「装飾アイコン」 など、崩れても読めるものだけに限定します。

4. 「下書き保存」禁止（One Source of Truth）

修正したいとき、ファイルの末尾に「修正用コード」を追記していく「継ぎ足し方式」は禁止です。

NG：同じファイル内で重複

既存のコードには触らず、一番下に新しいCSSを書いて上書きする。
.button { background: red; } (末尾に追記)
→ どこが有効なのか分からなくなり、管理不能になります。

OK：元の場所を直接修正

勇気を出して、元の記述を書き換える。
ファイル内検索（Ctrl+F）を使って、該当箇所の元の記述を探し出し、そこを直接修正します。

💡 AIへの指示のコツ
「既存の構造を壊さず、メンテナンス性を維持した美しいコードで出力してください」と一言添えるのがプロのディレクションです。

✨ AI時代のCSS設計

AI（Antigravity）と共同でデザインを磨き上げるための最新ルールです。

5. 脱・!important 戦略

OK

:where() セレクタを活用してベーススタイルを定義する。
詳細度（強さ）を上げすぎないことで、後の修正を楽にします。

6. デザイン・トークン（変数）の徹底

OK

色はすべて --renewal-gold のような変数で管理し、ハードコード（直書き）を排除します。
これにより、サイト全体のカラーチェンジが 1 箇所（:root）の修正で完了します。

7. コンポーネント指向メディアクエリ

OK

レスポンシブ設定は、そのクラスの本体のすぐ下に記述します。
PC版とスマホ版の設定を上下に往復して探す手間をなくします。

8. 動的パディング（clamp関数）

OK

padding: clamp(60px, 10vw, 120px) 0;
画面幅に合わせて余白が自動で「呼吸」するように設計します。

🚀 求人LP特化の3ヶ条

頻繁なABテストや修正が発生する「求人ランディングページ」で事故を防ぐための追加ルールです。

5. セクション＝積み木ルール（入れ替え自由自在）

求人LPは「先輩の声を上にしたい」「募集要項を最後にしたい」など、頻繁な並び替えが発生します。

OK

各セクションは上下に依存する余白(margin-top/bottom)を持たせない。
自分の内側の余白(padding)だけで完結させる。
→ これなら、どの順番に入れ替えてもレイアウトが崩れません。

6. 「文字・即・テキスト」 (脱・画像化)

給与、時間、待遇…これらは最も変更が多い箇所です。

NG

Photoshopで作った画像の中に文字を埋め込む。
→ 時給が50円変わるだけで画像の作り直しが発生します。

OK

HTMLテキスト＋CSSで作る。
変更の可能性がある情報は、意地でもテキストデータとして実装します。

7. CVボタンは絶対聖域

LPの唯一の目的は「応募」です。デザイン崩れでボタンが押せなくなることだけは万死に値します。

OK

最強の共通パーツを作る。
.btn-cta のような共通クラスを作り、どのページ、どの場所に置いても絶対に崩れないように独立させて管理します。

⚠️ 技術的負債を防ぐ

コードを書く前に知っておくべき、文字化けとファイル構造の罠

8. ファイルエンコーディングの厳守

【重要】PowerShellでファイルを保存する際の注意

問題

PowerShellの Set-Content や Out-File は、デフォルトでUTF-16 LEまたはシステムデフォルトエンコーディングで保存される。
これにより、日本語コメントを含むHTML/CSS/JSファイルが文字化けする。

OK（UTF-8 BOMなしを明示） 
[System.IO.File]::WriteAllText('file.css', $content, (New-Object System.Text.UTF8Encoding $false))

確認方法：
VS Codeの右下に表示されるエンコーディングが「UTF-8」であることを必ず確認する。

9. CSS設計：ファイル分割と直接修正

管理しやすくするためにファイルを分割しますが、分割されたファイルは直接修正します。

推奨構成（管理の単位） 
css/
├── base.css          （基本設定・リセット）
├── components.css    （ボタン、カードなど共通パーツ）
├── header.css        （ヘッダー専用）
├── hero.css          （ヒーローセクション専用）
├── sections.css      （各セクションの共通スタイル）
└── main.css          （上記を @import でまとめる）

修正の鉄則

  • 迷わず直接修正：header.css の中身を直接書き換える。
  • custom.css は「例外」：実験的な変更や、触れない外部ファイル（WPテーマ等）の場合のみ使用。
  • Gitが生命線：直接修正する代わりに、一歩ごとにコミットする（ルール10）。
メリット

✅ シンプル：1箇所見れば全てが分かる。
✅ 軽量：無駄な上書きコードが残らない。
✅ 確実：Gitがあれば失敗しても1秒で戻せる。

🔄 実践：無限修正ループからの脱出

今回のセッションで実際に起きた「試行錯誤→重複→文字化け」の悪循環を防ぐための必須ルール

10. Git必須（変更前に必ずコミット）

「ちょっと試してみる」が取り返しのつかない事態を招きます。

NG

いきなりファイルを編集し始める。
→ うまくいかなかった時、「元の状態」に戻せなくなる。

OK

変更前に必ずコミット

git add .
git commit -m "PRICE section 修正前"

# 実験的な変更はブランチで
git checkout -b fix/price-section

これで、いつでも git reset --hard HEAD で戻れます。

11. ワンステップ・ワンテスト（段階的テスト）

一度に複数の変更を加えると、どこで壊れたか分からなくなります。

NG：一気に変更

ボタンの幅 + 色 + フォント + 位置を同時に変更。
→ どれが原因でデザインが崩れたのか特定不可能。

OK：1つずつ確認
  1. ボタンの幅を変更 → ブラウザ確認
  2. OK → 色を変更 → ブラウザ確認
  3. OK → フォントを変更 → ブラウザ確認

問題が起きても、直前の1ステップだけ戻せばOK。

12. コマンド実行前チェックリスト

PowerShellでファイル操作をする前に、必ず以下を確認します。

実行前チェック
  • ☐ -Encoding UTF8 は指定したか？
  • ☐ バックアップは取ったか？（cp file.css file.css.bak）
  • ☐ 対象ファイルのパスは正しいか？
  • ☐ 上書きしても大丈夫なファイルか？（メインCSSではないか？）
  • ☐ Gitでコミット済みか？

13. ブラウザ確認を義務化

「コードが正しそう」と「実際に正しい」は別物です。

NG

コードを修正して満足。ブラウザで確認せずに次の作業へ。
→ 実は表示が崩れていた、ボタンが押せなかった、などが後で発覚。

OK：最低限の確認項目
  • ✅ デスクトップ表示（Chrome/Edge）
  • ✅ スマホ表示（DevTools: F12 → デバイスモード）
  • ✅ ボタンのクリック可否（実際にクリックしてみる）
  • ✅ テキストの読みやすさ（コントラスト、文字サイズ）

特に、スマホ表示は必須。ユーザーの8割はスマホです。

14. ロールバック手順の明文化

「やっぱり元に戻したい」時の手順を事前に決めておきます。

ロールバック手順

Git使用時：

# 最後のコミットに戻す
git reset --hard HEAD

# 特定のコミットに戻す
git reset --hard [コミットID]

# ファイル単位で戻す
git checkout HEAD -- file.css

バックアップファイル使用時：

# バックアップから復元
cp file.css.bak file.css

custom.css分離方式：

# custom.cssだけ削除すれば元に戻る
# メインCSSは無傷のまま
rm custom.css

15. AI（Antigravity）との協働ルール

AIが生成するコードやコマンドの特性を理解し、事故を未然に防ぎます。

環境を明示する
  • OS（Windows）、シェル（PowerShell）、エンコーディング（UTF-8）を常に意識させる。
  • ANTIGRAVITY_NOTES.md があれば活用し、AIにコンテキストを記憶させる。
コマンドの制約を伝える
  • PowerShell環境では grep ではなく Select-String を使うよう指示。
  • Linux系コマンド（sed, awk 等）は使えないことを明記する。
段階的に確認する
  • AIが生成したコードを盲信せず、必ずブラウザで確認（ルール13）を行う。
  • 一気に大量の修正を頼まず、セクションごとに段階的に依頼する。
カスタム指示の例 
{
  "antigravity.customInstructions": "PowerShell環境。UTF-8厳守。grep禁止→Select-String使用。"
}
