【技術調査レポート】
Windows Server 2019上のIISにおけるWebサーバー構築の是非と推奨環境について

本資料は、Windows Server 2019上のIIS（Internet Information Services）環境を利用してCMS（WordPressなど）およびWebサイトを構築・運用する場合の技術的なメリット・デメリット、および中長期的な安全運用に向けた代替構成について評価・調査した結果をまとめたものです。

1. IIS ＋ WordPressのメリット

IIS環境でWebシステムを運用する場合、以下の点がメリットとして挙げられます。

  • Microsoftエコシステムとの統合: 既存のWindows Server環境やActive Directory、SQL ServerなどのMicrosoft製品との親和性が高いです。
  • IISマネージャーによる管理: GUIベースの管理ツールであり、ウェブサイトの設定等を直感的に行えます。
  • Windows認証の利用: 企業内部ネットワーク環境においては、Windows認証を利用したアクセス制限が可能な場合があります。

※補足：これらのメリットは「大規模な企業（社員数千人レベル）が社内専用のシステム（イントラネット等）を構築する場合」にシステム管理側で大きく活きる仕様であり、今回のような外部の一般ユーザー向け公開用Webサイトにおいては、これらの機能的恩恵はほぼありません。

2. IIS ＋ WordPress の課題・致命的なデメリット

一方で、IISでWordPress（PHPを使用する動的システム）を運用する際には、以下の重大な運用リスクが指摘されています。

【重要】PHPサポートの公式終了

WordPressはPHP言語で動作しますが、Microsoftは「PHP 8.0以降のWindows版PHPの公式サポート」をすでに終了しています。現在WordPressを安全に稼働させるためにはPHP 8.3以上が推奨されていますが、公式のサポート（セキュリティパッチや緊急時のバグ修正）がない状態で非公式ビルドを使用し続けることは、システムを未知の脆弱性に晒し続けることを意味し、長期運用において極めて高いセキュリティリスクとなります。

パフォーマンス（動作速度）の低下

一般的に、WordPressはLinux系のサーバー（ApacheまたはNginx）で最適に動作するように設計されています。IIS環境では、PHPの実行構造の違いによりパフォーマンスが劣り、ページ表示速度の低下を招くケースが多数報告されています。

パーマリンク（URL設定）と互換性の問題

WordPressは通常「.htaccess」ファイルを使用してURL構造を制御しますが、IISではこのファイルが使用できないため「web.config」という別の複雑なファイルを代用する必要があります。これにより主流なプラグインが予期せぬエラーを起こすなど、設定と構造の複雑化によるトラブルシュートが非常に困難になります。

3. 結論と代替のソリューション提案

Windows Server 2019で動的システム（WordPressなど）を運用することは、PHPのサポート終了やセキュリティ・パフォーマンスの観点から推奨されません。
トラブルを防ぎ、サーバー保守の手間を最小限にするための解決策として、以下のいずれかの方法を強く推奨いたします。

  • 推奨A：専用のLinuxベースのサーバー（エックスサーバー等）の利用
    御社が現在運用されているように、管理のしやすさからWordPressなどの動的CMS環境を利用される場合は、Web運用に最適化されたLinuxベースのサーバー環境（エックスサーバー等）を別途用意して構築を行うことが、安定性とセキュリティの観点から最も無難で確実な選択です。
  • 推奨B：静的サイト（HTML出力）＋ ヘッドレスCMS構成の採用（※現在のWindowsサーバーを活用する場合）
    もし幼稚園側のご意向で現在のWindowsサーバーをどうしても継続利用する必要がある場合は、「PHP」や「データベース」等の環境構築を行わず、極めて安全な静的ファイル（HTML/CSS等）のみをそのまま配置して運用する方法を推奨します。
  • 【非推奨】プランC：Windowsサーバー内に仮想Linux環境（Docker / WSL2等）を構築してWordPressを運用
    様々な事情により「どうしても既存のWindows環境内でWordPressを稼働させたい」場合の、技術的な最終手段です。サーバー内部に仮想的にLinux環境を作り出し、その中でWordPressを動かします。技術的には可能ですが、インフラ構築の難易度が非常に跳ね上がり、システム構成も複雑化するため、サーバー管理者（運用保守担当者）様にかかる日常の管理負担や、エラー発生時の復旧コストが極めて大きくなってしまう点に強い注意が必要です。
